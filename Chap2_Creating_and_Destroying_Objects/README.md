# 2장. 객체 생성과 파괴

## 목표
1. 객체를 만들어야 할 때와 만들지 않아야 할 때 구분 (Item 1 ~ Item 4)
2. 올바른 객체 생성 방법과 불필요한 생성을 피하는 방법 (Item 5 ~ Item 7)
3. 제때 파괴됨을 보장하고 파괴 전에 수행해야 할 정리 작업을 관리하는 요령 (파괴: Destroy) (Item 8 ~ Item 9)

## Item 1. 생성자 대신 정적 팩터리 메서드를 고려하라

- 생성자 대신 **이름 있는 정적 메서드**(`from`, `of`, `valueOf`, `getInstance(instance)`, `newInstance(create)`, `getType`, `newType`, `type`)를 사용할 수 있다.
- **장점**
  - 의미 있는 이름으로 가독성이 향상된다.
  - 호출마다 새 객체를 만들지 않아도 된다. (캐싱/재사용이 가능하다.)
  - 반환 타입을 유연하게 변경할 수 있다. (**인터페이스 기반 설계에 유리하다.**)
    - 반환 타입의 하위 타입 객체도 포함된다.
    - 입력 매개변수에 따라 매번 다른 클래스의 객체를 반환할 수 있다.
  - 정적 팩터리 메서드를 작성하는 시점에는 반환할 객체의 클래스가 존재하지 않아도 된다. (ex) **JDBC**)
- **단점**
  - `public/protected` 생성자가 없으면 상속이 어렵기 때문에 하위 클래스 확장이 제한될 수 있다.
  - API 문서를 보지 않으면 발견이 어려울 수 있다.
 
## Item 2. 생성자에 매개변수가 많다면 빌더를 고려하라

- 선택 매개변수가 많을 때 점층적 생성자 패턴(가독성 하락)이나 JavaBeans 패턴의 단점(객체 하나를 생성하려면 메서드를 여러 개 호출해야 함, 객체가 완전히 생성되기 전까지는 일관성이 무너진 상태)을 해결할 수 있다.
- **빌더 패턴의 장점**
  - 가독성이 좋고, 객체 생성 코드가 명확하다.
  - 불변 객체를 만들기 쉽다.
  - 검증 로직을 `build()` 시점에 모아 처리 가능하다.
- 계층 구조 클래스에도 적용 가능하다. (재귀적 타입 한정 등)

## Item 3. private 생성자나 열거 타입으로 Singleton임을 보증하라

- **Singleton 구현 방법**
  - `public static final` 필드 방식
  - 정적 팩터리 방식
  - ⭐ 원소가 하나인 `enum` 방식 (권장: 왜냐하면 `enum` 방식은 직렬화/리플렉션 공격에 강하고 구현이 간결하기 때문이다)
- **단, Singleton이 꼭 필요한지 먼저 검토**! (이유: Singleton을 활용하는 클라이언트의 테스트는 어렵고, 전역 상태 문제도 있다)

## Item 4. 인스턴스화를 막으려거든 private 생성자를 사용하라

- 유틸리티(Utility) 클래스(`Math`, `Collections` 스타일)는 인스턴스화할 이유가 없다.
- 생성자를 `private`으로 선언해 명시적으로 인스턴스화를 차단한다. (**추상 클래스를 만드는 것만으로는 인스턴스화를 막을 수 없다.**)
- 필요하면 생성자 내부에서 `AssertionError`를 던져 실수 방지 의도를 강화한다.

## Item 5. 자원을 직접 명시하지 말고 의존 객체 주입(의존성 주입)을 사용하라

- ⭐ 클래스가 사용할 자원(ex) 사전, DB 커넥션, 정책 객체)을 내부에서 직접 말아라!
- ⭐ **생성자/팩터리/빌더를 통해 외부에서 주입받으면 아래와 같은 장점이 있다.**
  - **유연성이 증가**한다: 구현의 교체가 쉽다.
  - **테스트가 용이**하다: 목 객체 주입이 가능하다.
  - **재사용성 및 유지보수성**이 향상된다.
- 이 방법이 '정적 유틸 클래스 + 전역 자원' 구조보다 훨씬 바람직하다.

## Item 6. 불필요한 객체 생성을 피하라

- 같은 기능의 객체를 반복 생성하지 말고 재사용하라!
  - ex) `new String("abc")` 대신 문자열 리터럴
  - `Boolean.valueOf()` 같은 캐시 활용
- 무거운 객체(정규표현식 `Pattern` 등)는 한 번 만들어 재사용하기! (객체 생성 비용 문제가 존재한다.)
- 오토박싱으로 인한 불필요한 박싱 객체 생성 주의! (응답 시간의 매우 느려질 수 있다.)
- ⭐ 단, 과도한 최적화는 금물: 명확성을 해치지 않는 선에서 적용하는 것이 좋다.
- 그렇지만... 요즘 JVM의 Garbage Collector는 상당히 최적화가 잘 되어 있어서 작은 객체를 create하고 destroy하는 것은 괜찮으니 Database 연결 객체를 제외하고는 무리해서 객체 Pool을 만들 이유는 없다.

## Item 7. 다 쓴 객체 참조를 해제하라

- ⭐ **Java는 Garbage Collector가 있지만, 참조가 남아 있으면 회수되지 않아 메모리 누수 문제가 발생한다.**
- **메모리 누수 시 발생할 수 있는 문제점**
  - Garbage Collection 활동과 메모리 사용량이 늘어나 성능 저하
  - Disk Paging이나 `OutOfMemoryError`이 발생하여 프로그램의 비정상적인 종료 초래
- 특히 조심해야 할 부분: 직접 메모리를 관리하는 구조인 **`Stack` 클래스, Cache, Listener(Callback)**
- 따라서 원소를 제거할 때 참조를 null 처리해서 누수를 방지해야 한다.
- 약한 참조(`WeakHashMap`)나 만료 정책 등도 상황에 따라 활용하면 좋다.

## Item 8. finalizer(Java 8)와 cleaner(Java 8 이후) 사용을 피하라

- ⭐ **finalizer/cleaner는 실행 시점이 불확실하고 느리며 위험하다.**
  - 제때 실행되어야 하는 작업(ex) 파일 닫기)은 절대 할 수 없다.
  - 상태를 영구적으로 수정하는 작업에서는 절대로 활용하면 안된다.
  - 성능 문제도 동반된다.
  - 특히 finalizer의 경우 finalizer 공격에 노출되어 심각한 보안 문제를 일으킬 수 있다. 따라서 객체 생성을 확실하게 막으려면 Dummy용 `finalize` 메서드를 만들고 `final`로 선언하자!
- **중요한 자원 반납(ex) 파일, 소켓, DB 커넥션)은 명시적으로 처리해야 한다.
- ⭐ `AutoCloseable` + `try-with-resources`를 사용해야 한다.
- cleaner는 핵심 로직 대신 '안전망' 정도로만 제한적으로 고려하는 것이 좋다.

## Item 9. try-finally보다 try-with-resources를 사용하라
- `try-with-resources`는 자원 해제를 자동화하고 코드가 짧아 안전하다.
- 예외가 여러 개 발생해도 주요 예외/억제 예외를 더 잘 보존한다.
- 여러 자원도 선언부에서 깔끔하게 관리 가능하다.
- ⭐ **단, 자원 클래스가 AutoCloseable을 구현해야 한다는 전제가 존재한다.**
